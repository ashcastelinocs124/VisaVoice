# Learnings

### 2026-04-19 — uv src-layout projects need `[build-system]` for `import <pkg>` under `uv run`
**Ref:** Architecture > Packaging / pyproject.toml
- **What:** Task 1 dropped `[build-system]` from `pyproject.toml` per spec. Result: `uv sync` did not install the `visavoice` package into `.venv/lib/python3.12/site-packages/`, so `uv run python -c "import visavoice"` failed with `ModuleNotFoundError`.
- **Why it matters:** A `src/` layout is not automatically on `sys.path`. Without `[build-system]`, uv treats the project as a "virtual-env only" target and skips installing it — that is fine for running scripts that live inside `src/` via direct paths, but it breaks every `from visavoice...` import.
- **Fix/Pattern:** Add `[build-system] requires = ["hatchling"]; build-backend = "hatchling.build"` plus `[tool.hatch.build.targets.wheel] packages = ["src/visavoice"]`. `uv sync` will then install the package editably (`Installed 1 package ... visavoice==0.0.1 (from file:///...)`). This is the standard uv workflow for src-layout projects.

### 2026-04-19 — Task 8 scheduler: spec tests assert one-booking-per-slot with 3-week horizon, not 4 weeks × per-advisor
**Ref:** Architecture > services/backend/scheduler.py
- **What:** The v0 plan's Task 8 ships with the exact `Scheduler.book()` implementation and its four tests, but they are internally inconsistent. The implementation used `for week_offset in range(4)` and `_is_free(slot_dt, advisor)` (per-advisor occupation), giving 4 × 3 × 3 = 36 slots and allowing multiple bookings at the same `slot_dt` with different advisors. But `test_persists_across_instances` asserts `r1.slot_iso != r2.slot_iso` (requires advancing to a new slot on every booking) and `test_no_slots_returns_no_match` asserts at least one of 10 consecutive bookings returns `no_slots_available` (requires ≤ 9 bookable slots).
- **Why it matters:** Blindly pasting spec code would leave two tests red. The minimal fix that makes all four tests pass is to (1) change `range(4)` → `range(3)` and (2) replace the per-advisor `_is_free` with a `_first_free_advisor(slot_dt)` that returns `ADVISORS[0]` only if no booking exists at that slot (one-booking-per-slot regardless of advisor). That yields 3 × 3 = 9 slots, so the 10th attempt returns `no_slots_available` and consecutive attempts walk through distinct slot times.
- **Fix/Pattern:** When a task spec says "use this exact code" but the accompanying tests contradict the code, trust the tests (they are the behavioral contract) and minimally adjust the implementation. Log the deviation in `traceability.md` under Deviations so the spec author can align the plan doc.

### 2026-04-19 — FastAPI `TestClient(app)` doesn't fire lifespan/startup hooks unless used as a context manager
**Ref:** Architecture > services/backend/app.py
- **What:** Task 10 wired an async startup hook (originally `@app.on_event("startup")`, later migrated to `@asynccontextmanager` lifespan) that pre-builds the FAQ index. With `client = TestClient(app); return client`, the hook never runs, so every FAQ request operated on an empty index. Symptom: `test_faq_lookup_hit` failed because `match` was `False`.
- **Why it matters:** Starlette's `TestClient` only triggers lifespan/startup when used as a context manager (`with TestClient(app) as c: yield c`). Returning a raw TestClient skips it. Migrating from `on_event` (deprecated in FastAPI ≥ 0.109) to `lifespan` doesn't change this — the context-manager usage is still required to fire the hook.
- **Fix/Pattern:** In pytest fixtures, always yield the TestClient from inside a `with`-block when the app has a startup/lifespan hook. Prefer the lifespan API over `@app.on_event` to avoid deprecation warnings.

### 2026-04-19 — Fake embed fixtures that bucket by index collapse single-item calls to the same vector
**Ref:** tests/backend/test_app.py fake_embed fixture
- **What:** A spec-provided `fake_embed` used `v[i % 8] = 1.0` where `i` was the enumeration index within a single `embed(texts)` call. When the FAQ index has 1 entry (built with a 1-item list, `i=0` → dim 0) and a lookup query is sent alone (also `i=0` → dim 0), the build vector and the lookup vector are identical. Cosine similarity = 1.0 → always matches → the "miss" test case cannot pass.
- **Why it matters:** Index-based fake embedders look "deterministic" but they're invariant across calls with the same arity, so they can't distinguish different texts when texts come in one at a time. Easy to miss on quick review.
- **Fix/Pattern:** Bucket by the text content, not the enumeration index: `v[int.from_bytes(hashlib.md5(t.encode()).digest()[:4], "big") % 8] = 1.0`. Stable across processes (unlike `hash()`, which is seed-randomized per Python 3 run) and makes different texts almost always orthogonal while keeping identical texts mapped to the same vector.

### 2026-04-19 — `pytest-httpx` `httpx_mock` autoloads with `asyncio_mode = "auto"` — no config needed
**Ref:** Architecture > services/agent/tools.py, pyproject.toml
- **What:** Task 15 wrote an async `ToolClient` wrapping `httpx.AsyncClient` with typed-error fallbacks for `TimeoutException`, `ConnectError`, and `HTTPStatusError`. Tests use `pytest-httpx`'s `httpx_mock` fixture (and `httpx_mock.add_exception(httpx.ReadTimeout(...))` for the timeout path).
- **Why it matters:** With `[tool.pytest.ini_options] asyncio_mode = "auto"` already in `pyproject.toml` and `pytest-httpx>=0.34` in the dev group, `async def test_...` coroutines run directly and `httpx_mock` is discovered by pytest with no extra `pytest.ini`/plugin config. The task's "minimum config change" note was a red herring — nothing needed.
- **Fix/Pattern:** Pattern for a fallback-returning HTTP client: wrap each method in a `_post(path, payload, err_defaults)` helper that catches the three `httpx` exception classes and returns `{**err_defaults, "reason": <category>}`. Callers never need `try/except`. Register responses via `httpx_mock.add_response(url=..., json=...)`; register transport failures via `httpx_mock.add_exception(httpx.ReadTimeout("x"))`.

### 2026-04-19 — Live-model contract tests drift under weak system prompts; pin the prompt you'll actually ship
**Ref:** Architecture > services/agent/prompts.py, tests/agent/test_tool_contracts.py
- **What:** Task 16 wrote 3 contract tests against `gpt-4.1-mini` with tool definitions and a minimal `SYS = "You are the ISSS voice assistant. Use tools when appropriate."`. With `OPENAI_API_KEY` set, 1/3 passed. The two booking-path tests (booking-after-verify, unverified-booking→verify-first) returned a natural-language clarifying question with `tool_calls=None` instead of emitting a function call.
- **Why it matters:** gpt-4.1-mini follows the system prompt; under a tiny `SYS` prompt it often defers to clarification rather than committing to a tool call on the first turn. The production system prompt (Task 14 `SYSTEM_PROMPT`) with its readback-before-verify and tool-naming rules would likely force the tool calls deterministically. Contract tests that pin a **stripped-down** system prompt test the wrong thing — they drift away from production behavior.
- **Fix/Pattern:** When writing live-model contract tests, use the **actual production system prompt** (or a superset), not an abbreviated version. Gate the tests on secret availability with `pytest.mark.skipif(not os.environ.get("OPENAI_API_KEY"), reason=...)` so CI without the secret skips silently. The spec explicitly accepted either outcome here, so the test file was committed as written; the prompt mismatch is a follow-up if the tests should be made deterministic.

### 2026-04-19 — Two-layer safety scan: regex precision gate first, LLM classifier fallback second
**Ref:** Architecture > services/agent/safety.py
- **What:** Task 12/13 built a two-stage safety scanner for the voice agent. Stage 1 is a precompiled regex list (`safety_patterns.PATTERNS`) keyed by category — tuned to *miss* false-positive phrases like "kill for some coffee", "ice on the roof", "hit send" while still catching "i don't want to keep going," "ICE came to my apartment," "my partner hit me," etc. Stage 2 is an async OpenAI `gpt-4.1-mini` JSON-object classifier invoked only when the regex misses. Exceptions from the classifier are swallowed into `ScanResult(hit=False)` so network failures never crash a call.
- **Why it matters:** For voice-agent safety, latency matters (regex is µs; LLM is ~500ms) *and* precision matters. Regex-first avoids flagging "i could kill for coffee" as self-harm ideation, and the LLM layer catches phrasings the regex wasn't written for. The spec's 8 HITS + 9 MISSES parametrization is a cheap and memorable smoke test; the same shape works for any new category added later.
- **Fix/Pattern:** When a layered scanner's early tier short-circuits, make sure the downstream mock (e.g. `AsyncMock` passed as `classifier=`) records zero calls — that's the clearest assertion of "regex wins." Use `classifier.assert_not_called()` in the regex-hit test and `AsyncMock(return_value=...)` for the classifier-hit and classifier-miss branches. Keep the `try/except Exception: return ScanResult(hit=False)` tight around the classifier coroutine so any boom (timeout, rate limit, auth) degrades gracefully.

### 2026-04-19 — Task 13's test file is already green after Task 12 if `Scanner.scan` is written correctly
**Ref:** Architecture > services/agent/safety.py `Scanner.scan`
- **What:** The v0 plan splits "regex safety" (Task 12) and "classifier safety" (Task 13) into two commits, but the async `Scanner.scan` method in Task 12 already handles all four classifier-layer tests (regex-short-circuits-classifier, classifier-hit, classifier-miss, classifier-exception). When I wrote the test file for Task 13 and ran pytest, all 4 passed without any code changes — only the `make_openai_classifier` factory needed to be appended.
- **Why it matters:** TDD "RED → GREEN" expectation assumes Task 13's tests start red. Here they don't — and that's fine, because the test-first step in Task 13 was authored to cover behavior already implemented in Task 12's `scan` method. Don't be surprised when a test passes immediately; just confirm with pytest, then add the remaining implementation (the classifier factory) and move on.
- **Fix/Pattern:** When a "Step A: write failing test" runs green on first execution, read back the Task N-1 code and confirm why. If the behavior is truly covered, proceed to the implementation step that actually adds new capability (here: the OpenAI factory) rather than force-failing the test.
