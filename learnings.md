### 2026-04-19 â€” `# type: ignore` comments can mask real bugs, not just SDK stub gaps â€” verify the stubs first
**Ref:** Architecture > agent/main.py; CI > pyright configuration
- **What:** Code review of visavoice v0 caught a drain/shutdown inversion in `src/visavoice/agent/main.py`. The code had `session.drain()  # type: ignore[unused-coroutine]` (no await) and `await ctx.shutdown()  # type: ignore[general-type-issues]`. Both suppressions were rationalised as "LiveKit 1.5.4 stubs are wrong." In fact the stubs are correct: `asyncio.iscoroutinefunction(AgentSession.drain)` returns `True` (async â€” must await) and `asyncio.iscoroutinefunction(JobContext.shutdown)` returns `False` (sync â€” returns `None`, must NOT await). The unignored code would have produced (a) a coroutine warning "coroutine 'AgentSession.drain' was never awaited" for drain, and (b) `TypeError: object NoneType can't be used in 'await' expression` for shutdown. The `type: ignore` comments were silencing exactly the diagnostics that would have caught both bugs.
- **Why it matters:** Every safety-triggered call would have silently failed to drain pending turns AND crashed with a TypeError at the shutdown step, in a fail-open direction (the opposite of what a safety hook should do). A fire-and-forget `asyncio.create_task(_safety_handler(...))` with no done-callback swallowed the TypeError until GC time â€” no logs, no visible failure, just a worker that quietly kept going after a crisis utterance.
- **Fix/Pattern:** Before adding any `# type: ignore[...]`, verify the stub claim with `asyncio.iscoroutinefunction(Type.method)` (or `inspect.signature`). If the stub is correct, the type checker has caught a real bug â€” do NOT suppress. Also: always attach a done-callback to `asyncio.create_task(...)` for fire-and-forget coroutines in production paths, so exceptions are logged at task-completion time, not GC time. Extract the handler to a module-level function that takes its collaborators as arguments â€” it makes the code unit-testable with `AsyncMock`/`MagicMock` and the drain-vs-await distinction becomes a trivial assert (`mock.drain.assert_awaited_once()` vs `mock.shutdown.assert_called_once_with()`).

### 2026-04-19 â€” Duplicate `HIGH_SEVERITY` sets across modules: prefer one copy + "keep in sync" comment over cross-package import for independently-deployable services
**Ref:** Architecture > agent/safety.py, backend/escalation.py
- **What:** visavoice had two sets of "high-severity safety categories" â€” `agent/safety.py::HIGH_SEVERITY = {self_harm_ideation, acute_medical, abuse, deportation_threat, ice_contact}` and `backend/escalation.py::HIGH_SEVERITY_CATEGORIES = {..., deportation_threat}` (missing `ice_contact`). They diverged silently. Two fix options: (a) add `ice_contact` to the backend set, (b) hoist to a shared `visavoice.taxonomy` module imported from both.
- **Why it matters:** Option (b) introduces an agentâ†”backend import edge. The two packages are intended to be independently deployable (agent runs on LiveKit workers; backend runs as FastAPI on its own process), so keeping the import graph acyclic between them is more valuable than the DRY win. Option (a) is a one-line drop-in; the trade-off is that future additions need to be mirrored, which is an acceptable human-in-the-loop cost for two categories changing maybe twice a year.
- **Fix/Pattern:** When two "obviously shared" constants live in different deployment units, prefer a **duplicated constant + "keep in sync with X" comment** over a shared-module import, unless the constant changes often enough that drift risk dominates the coupling cost. Document the mirror relationship at the declaration site so the next reader doesn't reintroduce the coupling.

### 2026-04-19 â€” Adding ruff+pyright to an existing Python codebase: use per-file-ignores for prompts + tests, narrow `# type: ignore[rule]` for third-party SDK stub gaps
**Ref:** CI > .github/workflows/ci.yml; Architecture > agent/main.py, agent/prompts.py
- **What:** When wiring CI (Task 24) across the visavoice repo, the first `ruff check .` turned up 78 errors â€” 44 auto-fixable (import order I001, unused imports F401, `encode("utf-8")` UP012), and 34 that were either intentional (long lines in `agent/prompts.py`, long fixture strings in `tests/**`), or trivial real issues (`zip` without `strict=`, a `try/except/pass`, one-liner `for t in threads: t.start()`). `pyright src` turned up 2 type errors that were at first misread as LiveKit stub bugs on `session.drain()` / `ctx.shutdown()`. The stubs are actually correct â€” `drain` is async, `shutdown` is sync â€” and pyright was flagging a real drain/shutdown inversion in our own code. See the "drain/shutdown inversion" entry above.
- **Why it matters:** The knee-jerk fix of disabling E501 globally or adding a file-level `# pyright: reportUnknownMemberType=false` would hide future real issues in those files. The clean fix is targeted: `[tool.ruff.lint.per-file-ignores]` for the two files where long lines are semantically meaningful (system prompt + test fixtures), and two narrow single-line `# type: ignore[unused-coroutine]` / `# type: ignore[general-type-issues]` with one-line comments explaining the SDK stub mismatch. Keeps all the rules live everywhere else.
- **Fix/Pattern:** Run `ruff check . --fix` first; triage the remainder by code (`| awk '{print $1}' | sort | uniq -c | sort -rn`); for each remaining category decide: fix in-place, per-file-ignore (if structural), or narrow `# noqa`. For pyright SDK-stub issues, suppress the *specific rule* at the *specific line* with a comment explaining why it's a stub gap, never a blanket file-level suppression unless it's truly the whole file.

### 2026-04-19 â€” GitHub Actions `if:` expressions can reference secrets, but job-level gating is cleaner than step-level for secret-dependent suites
**Ref:** CI > .github/workflows/ci.yml
- **What:** Task 24's spec gates the tool-contract test suite on `OPENAI_API_KEY` availability using `if: ${{ secrets.OPENAI_API_KEY != '' }}` at the *job* level (not the step). Two jobs (`test`, `contracts`) with independent inputs and independent skip conditions is easier to reason about than one job with per-step gates â€” the PR check list shows a clean "contracts (skipped)" entry for forks that don't have the secret, and contributors don't wonder whether an individual step was silently skipped.
- **Why it matters:** Fork-safety on public repos: `secrets.*` is empty for PRs from forks, so any step that unconditionally tries to `${{ secrets.OPENAI_API_KEY }}` would pass an empty string to the test and produce confusing auth errors instead of a clean skip. Job-level `if` skips the entire job cleanly.
- **Fix/Pattern:** For any secret-dependent test suite, put it in its own job with `if: ${{ secrets.FOO != '' }}` at the job level. Keep the always-on job (lint + types + non-network tests) first so PR reviewers always see a green signal within ~60s.

### 2026-04-19 â€” livekit-agents 0.12.x â†’ 1.x renamed user/agent speech events and several APIs
**Ref:** Architecture > agent/main.py
- **What:** The visavoice plan was written for `livekit-agents 0.12.x` but `uv sync` resolved `livekit-agents==1.5.4`. Pasting the blueprint verbatim fails in several places. The deltas I hit while wiring Task 17:
  - `llm.function_tool` â†’ `livekit.agents.function_tool` (top-level, no longer under `llm`).
  - `agents.Agent(...)` â†’ `Agent(...)` directly from `livekit.agents`.
  - Event `user_speech_committed` (event.alternatives[0].text) â†’ `user_input_transcribed` emitting `UserInputTranscribedEvent(transcript: str, is_final: bool)`. Must filter `is_final=True` to only act on committed turns.
  - Event `agent_speech_committed` â†’ `conversation_item_added` emitting `ConversationItemAddedEvent(item: ChatMessage | AgentHandoff)`. Use `item.role == "assistant"` + `item.text_content`.
  - `JobContext.add_shutdown_callback` is a method that returns `None` in 1.x, not a decorator factory. Using `@ctx.add_shutdown_callback` assigns `None` back to the callback name â€” call it as `ctx.add_shutdown_callback(fn)` instead.
  - `AgentSession.drain()` is **async** in 1.x (must be `await`ed). `JobContext.shutdown()` is **sync** (returns `None`, must NOT be `await`ed). An earlier version of this file said the opposite â€” that was wrong and was caught in code review (see the 2026-04-19 entry on the drain/shutdown inversion below). `AgentSession.interrupt()` returns an awaitable future.
- **Why it matters:** Every one of these would have been a runtime AttributeError or a silent no-op (the shutdown hook being the most dangerous: it would look wired up but the callback would never run). Worse, the import-only verification gate passes for all of them except the decorator-as-method one, so static checks do not catch it.
- **Fix/Pattern:** When the plan's SDK version doesn't match the installed version, introspect the installed surface directly â€” `uv run python -c "import X; print(dir(X))"` and `inspect.signature(...)` â€” rather than trusting the blueprint. Capture the version mismatch in traceability.md's Deviations section with the exact 0.12.x â†’ 1.x mapping so a follow-up maintainer doesn't have to rediscover it.

### 2026-04-19 â€” uv src-layout projects need `[build-system]` for `import <pkg>` under `uv run`
**Ref:** Architecture > Packaging / pyproject.toml
- **What:** Task 1 dropped `[build-system]` from `pyproject.toml` per spec. Result: `uv sync` did not install the `visavoice` package into `.venv/lib/python3.12/site-packages/`, so `uv run python -c "import visavoice"` failed with `ModuleNotFoundError`.
- **Why it matters:** A `src/` layout is not automatically on `sys.path`. Without `[build-system]`, uv treats the project as a "virtual-env only" target and skips installing it â€” that is fine for running scripts that live inside `src/` via direct paths, but it breaks every `from visavoice...` import.
- **Fix/Pattern:** Add `[build-system] requires = ["hatchling"]; build-backend = "hatchling.build"` plus `[tool.hatch.build.targets.wheel] packages = ["src/visavoice"]`. `uv sync` will then install the package editably (`Installed 1 package ... visavoice==0.0.1 (from file:///...)`). This is the standard uv workflow for src-layout projects.

### 2026-04-19 â€” Task 8 scheduler: spec tests assert one-booking-per-slot with 3-week horizon, not 4 weeks Ă— per-advisor
**Ref:** Architecture > services/backend/scheduler.py
- **What:** The v0 plan's Task 8 ships with the exact `Scheduler.book()` implementation and its four tests, but they are internally inconsistent. The implementation used `for week_offset in range(4)` and `_is_free(slot_dt, advisor)` (per-advisor occupation), giving 4 Ă— 3 Ă— 3 = 36 slots and allowing multiple bookings at the same `slot_dt` with different advisors. But `test_persists_across_instances` asserts `r1.slot_iso != r2.slot_iso` (requires advancing to a new slot on every booking) and `test_no_slots_returns_no_match` asserts at least one of 10 consecutive bookings returns `no_slots_available` (requires â‰¤ 9 bookable slots).
- **Why it matters:** Blindly pasting spec code would leave two tests red. The minimal fix that makes all four tests pass is to (1) change `range(4)` â†’ `range(3)` and (2) replace the per-advisor `_is_free` with a `_first_free_advisor(slot_dt)` that returns `ADVISORS[0]` only if no booking exists at that slot (one-booking-per-slot regardless of advisor). That yields 3 Ă— 3 = 9 slots, so the 10th attempt returns `no_slots_available` and consecutive attempts walk through distinct slot times.
- **Fix/Pattern:** When a task spec says "use this exact code" but the accompanying tests contradict the code, trust the tests (they are the behavioral contract) and minimally adjust the implementation. ### 2026-04-19 â€” FastAPI `TestClient(app)` doesn't fire lifespan/startup hooks unless used as a context manager
**Ref:** Architecture > services/backend/app.py
- **What:** Task 10 wired an async startup hook (originally `@app.on_event("startup")`, later migrated to `@asynccontextmanager` lifespan) that pre-builds the FAQ index. With `client = TestClient(app); return client`, the hook never runs, so every FAQ request operated on an empty index. Symptom: `test_faq_lookup_hit` failed because `match` was `False`.
- **Why it matters:** Starlette's `TestClient` only triggers lifespan/startup when used as a context manager (`with TestClient(app) as c: yield c`). Returning a raw TestClient skips it. Migrating from `on_event` (deprecated in FastAPI â‰Ą 0.109) to `lifespan` doesn't change this â€” the context-manager usage is still required to fire the hook.
- **Fix/Pattern:** In pytest fixtures, always yield the TestClient from inside a `with`-block when the app has a startup/lifespan hook. Prefer the lifespan API over `@app.on_event` to avoid deprecation warnings.

### 2026-04-19 â€” Fake embed fixtures that bucket by index collapse single-item calls to the same vector
**Ref:** tests/backend/test_app.py fake_embed fixture
- **What:** A spec-provided `fake_embed` used `v[i % 8] = 1.0` where `i` was the enumeration index within a single `embed(texts)` call. When the FAQ index has 1 entry (built with a 1-item list, `i=0` â†’ dim 0) and a lookup query is sent alone (also `i=0` â†’ dim 0), the build vector and the lookup vector are identical. Cosine similarity = 1.0 â†’ always matches â†’ the "miss" test case cannot pass.
- **Why it matters:** Index-based fake embedders look "deterministic" but they're invariant across calls with the same arity, so they can't distinguish different texts when texts come in one at a time. Easy to miss on quick review.
- **Fix/Pattern:** Bucket by the text content, not the enumeration index: `v[int.from_bytes(hashlib.md5(t.encode()).digest()[:4], "big") % 8] = 1.0`. Stable across processes (unlike `hash()`, which is seed-randomized per Python 3 run) and makes different texts almost always orthogonal while keeping identical texts mapped to the same vector.

### 2026-04-19 â€” `pytest-httpx` `httpx_mock` autoloads with `asyncio_mode = "auto"` â€” no config needed
**Ref:** Architecture > services/agent/tools.py, pyproject.toml
- **What:** Task 15 wrote an async `ToolClient` wrapping `httpx.AsyncClient` with typed-error fallbacks for `TimeoutException`, `ConnectError`, and `HTTPStatusError`. Tests use `pytest-httpx`'s `httpx_mock` fixture (and `httpx_mock.add_exception(httpx.ReadTimeout(...))` for the timeout path).
- **Why it matters:** With `[tool.pytest.ini_options] asyncio_mode = "auto"` already in `pyproject.toml` and `pytest-httpx>=0.34` in the dev group, `async def test_...` coroutines run directly and `httpx_mock` is discovered by pytest with no extra `pytest.ini`/plugin config. The task's "minimum config change" note was a red herring â€” nothing needed.
- **Fix/Pattern:** Pattern for a fallback-returning HTTP client: wrap each method in a `_post(path, payload, err_defaults)` helper that catches the three `httpx` exception classes and returns `{**err_defaults, "reason": <category>}`. Callers never need `try/except`. Register responses via `httpx_mock.add_response(url=..., json=...)`; register transport failures via `httpx_mock.add_exception(httpx.ReadTimeout("x"))`.

### 2026-04-19 â€” Two-layer safety scan: regex precision gate first, LLM classifier fallback second
**Ref:** Architecture > services/agent/safety.py
- **What:** Task 12/13 built a two-stage safety scanner for the voice agent. Stage 1 is a precompiled regex list (`safety_patterns.PATTERNS`) keyed by category â€” tuned to *miss* false-positive phrases like "kill for some coffee", "ice on the roof", "hit send" while still catching "i don't want to keep going," "ICE came to my apartment," "my partner hit me," etc. Stage 2 is an async OpenAI `gpt-4.1-mini` JSON-object classifier invoked only when the regex misses. Exceptions from the classifier are swallowed into `ScanResult(hit=False)` so network failures never crash a call.
- **Why it matters:** For voice-agent safety, latency matters (regex is Âµs; LLM is ~500ms) *and* precision matters. Regex-first avoids flagging "i could kill for coffee" as self-harm ideation, and the LLM layer catches phrasings the regex wasn't written for. The spec's 8 HITS + 9 MISSES parametrization is a cheap and memorable smoke test; the same shape works for any new category added later.
- **Fix/Pattern:** When a layered scanner's early tier short-circuits, make sure the downstream mock (e.g. `AsyncMock` passed as `classifier=`) records zero calls â€” that's the clearest assertion of "regex wins." Use `classifier.assert_not_called()` in the regex-hit test and `AsyncMock(return_value=...)` for the classifier-hit and classifier-miss branches. Keep the `try/except Exception: return ScanResult(hit=False)` tight around the classifier coroutine so any boom (timeout, rate limit, auth) degrades gracefully.
